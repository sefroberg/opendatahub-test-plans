# TC-E2E-004: Multi-tenant per-request API key override

**Objective**: Validate the multi-tenant scenario where different
users send concurrent requests with different per-request API
keys via `x-ogx-provider-data` header.

**Preconditions**:

- OGX distribution deployed with `remote::gemini` provider active
- Primary `GEMINI_API_KEY` configured in the distribution
- Two secondary valid Gemini API keys available

**Test Steps**:

1. Send three concurrent POST requests to
   `/v1/chat/completions`:
   - Request A: no `x-ogx-provider-data` (uses config key)
   - Request B: `x-ogx-provider-data` with secondary key 1
   - Request C: `x-ogx-provider-data` with secondary key 2
2. Wait for all three responses
3. Verify all three succeed independently
4. Verify no cross-contamination between keys (each request
   authenticates with its own key)

**Expected Results**:

- All three requests return HTTP 200 with valid chat
  completion responses
- Each request authenticates with its respective API key
- No cross-contamination between concurrent requests
- The config-level key is unaffected by per-request overrides
